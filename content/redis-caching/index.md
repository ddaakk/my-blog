## 🟢 읽기 전략 (Read Strategies)
읽기 전략은 데이터 조회 시 캐시와 데이터베이스 사이의 데이터 처리 방식을 결정합니다.

### Read-Through
클라이언트가 데이터를 요청하면 캐시를 먼저 확인하고, 캐시에 데이터가 없으면 캐시가 자동으로 데이터베이스에서 데이터를 가져와 저장한 후 클라이언트에 반환합니다.

- **장점**
    - 캐시 관리 자동화: 캐시 시스템이 데이터베이스와 통합되어 캐시 미스를 처리합니다.
    - 높은 캐시 히트율: 자주 사용되는 데이터가 캐시에 유지되어 읽기 성능이 향상됩니다.

- **단점**
    - 초기 지연 발생: 처음 데이터 요청 시 캐시 미스로 인해 응답 시간이 길어질 수 있습니다.
    - 복잡한 설정 필요: 캐시와 데이터베이스 간 통합을 지원해야 합니다.

### Cache-Aside (Lazy Loading)
애플리케이션이 직접 캐시를 관리하며, 데이터를 읽을 때 캐시에 없으면 데이터베이스에서 가져와 캐시에 저장하고 반환합니다.

- **장점**
    - 유연한 캐시 관리: 개발자가 캐시 로직을 직접 구현하여 세부적인 제어가 가능합니다.
    - 캐시 효율성 증대: 자주 사용하는 데이터만 캐시에 저장하므로 메모리 사용을 최적화할 수 있습니다.

- **단점**
    - 개발 복잡도 증가: 캐시 관리 로직을 직접 구현해야 하므로 개발 및 유지보수가 복잡해질 수 있습니다.
    - 초기 캐시 미스: 캐시에 데이터가 없을 때 응답 시간이 길어질 수 있습니다.

### Refresh-Ahead
데이터 만료 전에 미리 캐시를 갱신하여 캐시 미스를 줄이는 전략입니다. 예상되는 데이터 요청에 대비해 백그라운드에서 캐시를 업데이트합니다.

- **장점**
    - 일관된 응답 시간: 캐시 미스를 방지하여 안정적인 성능을 제공합니다.

- **단점**
    - 리소스 낭비 가능성: 사용되지 않을 데이터를 갱신하여 불필요한 부하가 발생할 수 있습니다.
    - 예측 실패 위험: 잘못된 데이터 접근 패턴 예측으로 효율성이 떨어질 수 있습니다.

### TTL (Time to Live) 기반 캐싱
캐시에 저장된 데이터에 유효 기간(TTL)을 설정하여 일정 시간이 지나면 자동으로 삭제되게 하는 방식입니다.

- **장점**
    - 데이터 최신성 유지: 오래된 데이터가 자동으로 제거되어 최신 데이터를 제공할 수 있습니다.
    - 메모리 관리 용이: 불필요한 데이터가 제거되어 메모리 사용을 최적화합니다.

- **단점**
    - 적절한 TTL 설정 어려움: TTL이 너무 짧으면 캐시 미스가 증가하고, 너무 길면 오래된 데이터가 유지될 수 있습니다.
    - 캐시 미스 증가 가능성: 데이터가 만료되면 다시 데이터베이스에서 가져와야 하므로 응답 지연이 발생할 수 있습니다.

## 🔵 쓰기 전략 (Write Strategies)

쓰기 전략은 데이터 변경 시 캐시와 데이터베이스 사이의 데이터 처리 방식을 결정합니다.

### Write-Through (쓰기 직행)
데이터를 캐시와 데이터베이스에 동시에 기록하는 방식입니다. 클라이언트가 데이터를 쓰면 Redis 캐시와 데이터베이스에 모두 즉시 반영됩니다.

- **장점**
    - 데이터 일관성이 높습니다. 캐시와 데이터베이스의 데이터가 항상 동기화되어 있습니다.
    - 읽기 성능 향상: 최신 데이터가 캐시에 있어 빠르게 접근할 수 있습니다.

- **단점**
    - 쓰기 지연: 캐시와 데이터베이스에 동시에 쓰기 작업을 수행하므로 쓰기 속도가 느려질 수 있습니다.

### Write-Around (쓰기 우회)
데이터를 데이터베이스에만 쓰고 캐시에는 쓰지 않는 방식입니다. 이후 해당 데이터를 읽을 때 캐시에 로드됩니다.

- **장점**
    - 캐시 효율성 증대: 자주 사용되지 않는 데이터로 캐시가 오염되는 것을 방지하여 캐시 공간을 효율적으로 사용합니다.

- **단점**
    - 초기 읽기 지연: 처음 데이터를 읽을 때 캐시에 없으므로 데이터베이스에서 가져와야 하며, 이로 인해 지연이 발생할 수 있습니다.

### Write-Back (쓰기 지연)
데이터를 먼저 캐시에 기록하고 나중에 데이터베이스에 반영하는 방식입니다. 캐시에 쓴 데이터는 일정 조건(예: 시간 간격, 배치 처리 등)에 따라 데이터베이스에 쓰여집니다.

- **장점**
    - 쓰기 성능 향상: 쓰기 작업이 주로 캐시에서 이루어지므로 빠른 응답이 가능합니다.
    - 데이터베이스 부하 감소: 데이터베이스에 대한 쓰기 작업 빈도가 줄어듭니다.

- **단점**
    - 데이터 일관성 문제: 캐시와 데이터베이스 간 데이터 불일치가 발생할 수 있으며, 캐시 장애 시 데이터 손실 위험이 있습니다.
    - 복잡한 구현 필요: 데이터 동기화를 위한 추가 로직이 필요합니다.
